# 订阅节点串行批量检测设计

**目标**

把订阅节点弹窗里的“检测全部”改成可感知进度的串行检测：按节点逐个执行，每执行到一个节点时，该节点所在行的“检测”按钮进入加载态，便于观察当前跑到哪个节点。

**范围**

- 订阅节点弹窗中的“检测全部”改为前端串行调度。
- 批量检测过程中：
  - 顶部“检测全部”按钮保持加载态。
  - 当前检测节点对应的行内按钮显示“检测中...”。
  - 每个节点完成后，立即更新该行的可用性结果。
  - 整个过程不弹成功提示；只有请求异常时弹错误提示。
- 后端订阅节点 bulk 检测语义同步收敛为串行，避免与前端行为不一致。

**不做**

- 不改实例列表的检测流程。
- 不引入 SSE、WebSocket 或流式 HTMX 返回。
- 不改单节点检测接口语义。

**根因**

- 当前“检测全部”是一次性请求 `/api/subscriptions/:id/proxies/check`，后端跑完整批后才返回整窗 HTML。
- 前端只知道“开始”和“结束”，不知道中间进度，自然无法把某一行按钮切到“检测中”。
- 即使此前把 bulk 并发限制收紧，用户仍然只能看到最终结果，无法定位当前执行位置。

**方案**

- `ui_htmx.go`
  - 给订阅节点弹窗补充前端串行检测所需的数据标记：
    - 顶部“检测全部”按钮改为前端触发，不再直接 `hx-post` bulk UI action。
    - 节点行增加精确节点名、健康状态单元格、单节点检测 URL 等数据属性。
- `htmx.js`
  - 新增订阅节点 bulk 串行检测编排函数：
    - 读取当前弹窗内节点行顺序。
    - 逐个调用 `/api/subscriptions/:id/proxies/check` 的单节点检测。
    - 在每轮请求前后切换对应行按钮加载态，并实时更新健康状态单元格。
    - 若弹窗关闭或节点 DOM 被移除，则停止后续检测。
  - bulk 执行期间禁用其他行内检测按钮，避免并发交叉。
- `app.go`
  - 订阅节点 bulk 检测并发限制统一改为串行 `1`，保持 API 语义与 UI 一致。
- `style.css`
  - 复用现有按钮 loading 样式，不新增复杂视觉状态；必要时补一个 bulk 期间禁用态。

**验收标准**

- 点击“检测全部”后，按表格顺序逐行检测。
- 任一时刻只有当前节点的“检测”按钮显示“检测中...”。
- 当前节点检测完成后，该行可用性结果立即刷新。
- 检测过程中弹窗滚动位置保持，不回到顶部。
- 检测过程中不弹成功提示；请求异常时才弹错误提示。
- 订阅节点 bulk 检测后端语义为串行。
